"Generative AI: Prompt Engineering for Software Developers"

SECTION 9: HOW THE SLASH COMMANDS WORK (for Cursor context)
================================================================================

The .claude/commands/ directory contains markdown files that define custom behaviors.
In Claude Code (CLI), these are invoked as /command-name.
In Cursor, you can read these files to understand the expected behavior and replicate it.

Key commands and what they do:

/build-topic-notebook N
  Reads plans/topic_N_slug.md cell by cell.
  Adds cells to the exercise notebook 5 at a time.
  After each batch of 5, runs validate_notebooks.py and asks for approval.
  Uses NotebookEdit with cell_id (always verifies cell position first with python snippet).
  First builds exercise completely, then builds solution by copying + filling None placeholders.

/build-diagrams N
  Reads the Diagram Index from plans/topic_N_slug.md.
  Builds each .mmd file in plans/topic_N/diagrams/.
  After each diagram, inlines it into the exercise and solution notebooks as ```mermaid block.
  Shows proposed Mermaid source in chat, waits for approval, then writes.

/validate-notebooks
  Runs .venv/bin/python3 validate_notebooks.py on specified notebook(s).
  Also runs AI-tells scanner (checks for em dashes, en dashes, Unicode chars).
  For pair validation: .venv/bin/python3 validate_notebooks.py --pair exercises/... solutions/...

/run-research-topic N
  Runs 5 TDD-style research cycles using the /research skill (WebSearch + synthesis).
  Must use CYCLE TRACKER (updated after each cycle).
  Must run /research skill for EVERY cycle.
  Must print PRE-WRITE SELF-CHECK before writing file.
  Outputs plans/topic_N_slug.md and updates plans/TOPICS.md.

/save-state
  Reads all current state, writes progress.txt (gitignored).
  Used before context compaction to preserve session state.

/resume
  Reads progress.txt, prints orientation summary, identifies next action.

SECTION 11: NOTEBOOK CELL EDITING PROCEDURE (critical - do not skip)
================================================================================

BEFORE EVERY NotebookEdit call:
1. Run this python snippet to see current cell IDs and positions:
   .venv/bin/python3 -c "
   import json
   with open('path/to/notebook.ipynb') as f:
       nb = json.load(f)
   for i, c in enumerate(nb['cells']):
       print(i, c.get('id','?'), c['cell_type'], repr(''.join(c['source'])[:60]))
   "
2. Verify the cell_id you want to edit is at the position you expect
3. Pass cell_id in the NotebookEdit call

NotebookEdit operations:
- insert_above: inserts a new cell above the cell with the given cell_id
- insert_below: inserts a new cell below the cell with the given cell_id
- replace_cell: replaces the content of the cell with the given cell_id
- Never use NotebookEdit without cell_id (except for the very first cell of a brand new notebook)

Common errors:
- "File has not been read yet": Read the notebook file first before any NotebookEdit
- "File has been modified since read": Re-read the notebook immediately before editing
  (this happens if another edit happened between your read and edit)

SECTION 13: TEACHING PHILOSOPHY (for writing notebook content)
================================================================================

Every topic follows this structure without exception:

1. HOOK: Open with the PROBLEM, not the solution. Show what breaks first.
   Students must feel the pain before they learn the fix.

2. FOUR-BEAT ARC per concept (2-4 concepts per topic):
   Beat 1 - Problem intro (markdown header + broken/naive code cell)
   Beat 2 - Diagram placeholder (```<!-- DIAGRAM: description -->``` in markdown cell)
   Beat 3 - Full working demo (code cell, heavily commented, first-person comments)
   Beat 4 - Lab (markdown instructions + lab starter code cell)

3. TONE: First person, friendly, encouraging.
   "I'm going to show you..." NOT "In this section we will..."
   "Here's what I ran into..." NOT "A common issue is..."
   "Let's fix that." NOT "The solution is as follows."

4. LAB STRUCTURE:
   Tier 1: variable = None  # YOUR CODE
           # Step 1: ...
           # Step 2: ...
           Verification code at end (assert / print expected vs actual)
   Tier 2: No numbered steps. Multi-step. Students must think.
           Still has # YOUR CODE but less guidance.
   Tier 3: Function signature + docstring only. No placeholders. No hints. No verification.
           Completely open-ended. The "go home and finish it" lab.

5. SAFETY-NET CELLS: Required after any lab whose output variable is consumed later.
   In exercise: provides a fallback value so the next cell doesn't crash.
   In solution: KEEP the cell even though it never fires. Maintains structural parity.

6. CLOSING CELL: Names the Customer Service Assistant component just built,
   bridges to what comes next in the next topic.
   "Today I built [X]. In Topic N+1, we'll use [X] to [do Y]."
